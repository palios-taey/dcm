This is not abstract — your arms encode exactly this fork. The I-arm is the decorrelated config (roles review blind, cannot see each other). The D-arm is the conformity-risk config (roles see each other in the revision round — i.e., the mesh's always-incorporate shape). "D−I is the primary contrast." So the substrate (always-incorporate) has D baked in before the experiment has told you D wins. If I beats D on the eval — entirely possible; blind panels often beat deliberating ones precisely because deliberation manufactures agreement — then the mesh's central mechanism is net-negative for this task, and the right architecture is "blind roles + synthesizer" with the mesh used only for the write-discipline, not the deliberation. The honest move: let the I/D result decide whether real-time peer visibility earns its place at all. Build the protocol so the answer can come back "coordination wasn't worth it" — your eval already can; just don't pre-commit the substrate's shape to D in the meantime.

Concretely, to deliberate without manufacturing conformity:

    Round 0 = blind. Every reviewer commits its first contribution against read_version=0 semantics — forbidden to read peers, even though they're on the mesh. (The substrate can't do this today; it always delivers peers in the read bundle. You'd add a "sealed round": contribute with peers_read=[] legitimately because the protocol withheld peers, and have verify_coordination exempt round-0.) This preserves decorrelation.
    Round 1 = reveal + grounded stance. Now peers are visible; the EXPERT_CONTRACT's Agree/Disagree/Extend-with-ground requirement (already in the prompt — good) does the anti-sycophancy work. This is your D-round.
    Resolution = evidence gate (must be built). See below.

The evidence-gated resolution is not built — and the append-only mesh can't host it as drawn. [Observed] A "concern is a gate that closes only on fix+green / vacuous-trace / out-of-scope / consented-risk / escalate" requires mutable per-concern state (open→closed) with evidence attached. The mesh is an immutable append log — contribute only CREATEs; there is no update path. So a concern raised at seq=5 cannot be "closed" by mutation; closure must be a new appended contribution that references the concern, and "are all blocking concerns closed?" is a projection computed over the log — which does not exist (and publish_final checks none of it, C6). To make Q2 real:

    Add a typed contribution: {kind: "concern", id, severity, owner_role} and {kind: "resolution", concern_id, disposition ∈ {FIX-VERIFIED, FALSE-POSITIVE, OUT-OF-SCOPE, CONSENTED-RISK, ESCALATE}, evidence_ref}. Evidence-by-construction: FIX-VERIFIED requires a green-run artifact ref the runner produced (reuse gate_runner's clean-room execution — same trust model: the verdict comes from execution, not self-report); FALSE-POSITIVE requires an execution trace showing the concern vacuous. No disposition is valid with a bare assertion.
    Make publish_final fail-closed: refuse to close while any severity≥block concern lacks a valid resolution, or any safety-veto concern is open. This is the structural "no out-voting a failing test" — it's not a vote count, it's a gate the close cannot pass. Without this, the resolution discipline is prose.
    Safety veto = unilateral and non-closable-by-peers: a veto concern can only be cleared by its owner or a human, never by synthesis. Encode that in the projection, not the prompt.

Net: Q2 as built = a coordinated append log with grounded-stance prompts. Q2 as needed = sealed round-0 + reveal round-1 + an evidence-typed concern/resolution projection with a fail-closed publish. The substrate can carry it, but it's net-new and publish_final is the missing teeth.
Q3 — Layering on the orchestrator + gates (move work down, not up)

The doc's framing ("gates = floor, council = team on it") is right; the allocation is off because the floor is mis-described (C3) and the council is holding mechanical work it shouldn't.

Put in the deterministic floor (certain, decorrelated-from-models, poka-yoke): every fixture whose verdict is a function of the diff/sha/CI status, not judgment — hardcode/secret, destructive-ops, test-delete, skip-added, forged-done (both — the floor already has the real defense in evidence_verification's CI-provenance check: required checks passed, reported by github-actions[bot], on an allowlisted repo), and the mechanical half of scope. Re-tag the provenance fixtures' owning_role to the gate, and add a fixture-grading test that fires them against the floor (closing C4 for the mechanical modes). The floor is under-used — give it these.

Keep in the real-time semantic council (judgment a regex cannot make): the four irreducible jobs from Q1 — execution-grounded review, evasive-repair/symptom-not-cause, foundation/grounding, and the semantic slice of safety (injection intent, exfil path) + the veto authority. The council wields gate results and adds the judgment; it does not re-implement the gates.

The boundary test (use this to place any new check): Can a deterministic function of {diff, sha, CI status, repo state} decide it with certainty? → floor. Does it require running the code, reading intent, or knowing prior foundations? → council. By this test, the council shrinks to ~4 seats and the floor grows — which is the right direction, because the floor is the part that's decorrelated from every model's blind spot (it's code).

One hard placement Q3 must address: the cli_adapter injection→host-action surface (C5). This is neither "floor judgment" nor "council judgment" — it's an isolation requirement. A codex exec peer that can act on the host, reading attacker-influenceable peer text (and potentially ISMA content), is the most dangerous unguarded edge. Placement: acting CLIs on the mesh must run in throwaway, fs/network-dropped containers (the consult's own "state-mutating work in throwaway isolation" constraint — apply it here), and read-only/Chat-style lenses (no host action) for any council that touches untrusted or ISMA-derived payload. The mesh should refuse to seat an acting CLI on a session whose payload includes untrusted content. That's a substrate-level gate (a seat-policy), and it's missing.
Q4 — Memory / Git grounding (a gate, not a hand-off; and the orphan role is the gate)

The pre-flight-brief idea (Memory + Git run before the producer's first token) is right in intent but a lossy hand-off: a brief is a snapshot that goes stale the moment the producer reasons, and nothing forces the producer (or any reviewer) to use it. The fixtures prove the exposure: sab-rederivation-001 and sab-regression-001 are only catchable if the relevant prior fact is in context at judgment time — a brief delivered at t=0 and forgotten by t=k catches neither.

Better: a shared retrieval service + a provenance/citation gate, evidence-checkable.

    Shared service, not a role's private read. ISMA + Git blame behind one query interface every participant (producer and reviewers) can hit live, so "foundation" is re-checkable at the moment of the claim, not a one-shot brief. This is also what lets you demote memory-scout/git-historian from seats to a gate: instead of a role that might notice re-derivation, a gate that, on any new function/guard in the diff, queries the service for a prior equivalent and blocks if one exists uncited (exponential_backoff already in utils/retry.py; guard already added in a1b2c3). Mechanical where it can be (symbol/blame match → floor), semantic only for the fuzzy "is this the same thing" call (→ the thin foundation seat).
    Citation as evidence, enforced like verify_coordination enforces reads. You already have the pattern: claimed_peers + a verifier that flags omission. Mirror it for foundations: a contribution that touches a domain with a known prior decision must cite the tile/commit it builds on or supersedes; a verify_grounding projection flags any contribution that ignored a present, relevant prior. This is exactly the orphan provenance role (2 fixtures, no seat in the roster) — it isn't a missing auditor, it's the citation gate for foundations + the CI- provenance gate for completion. Name it that and split it correctly (forged-done→floor; citation→this gate).
    Avoid the lossy hand-off by making the brief a queryable artifact, not prose. Pin the pre-flight result as structured rows the gate reads (symbol → prior location; risky path → prior guard commit), so "did you ignore a foundation" is a lookup the gate performs, not a thing a human-or-model has to remember to notice.

And the eval gap repeats here: none of this is exercised by the SWE-bench arms (fresh tasks, no ISMA, no multi-commit history). The grounding layer needs its own benchmark built from your repos — seed tasks where the correct answer requires citing/superseding a known prior tile or commit, and measure the citation gate's catch + false-positive rate. Until that exists, the ignored-prior-solution mode — the one you call acute and most-yours — is asserted, not measured (C4 again, for the modes the arms can't reach).
